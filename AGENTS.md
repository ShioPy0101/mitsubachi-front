# ファイル管理サービスのUI全面実装・改善指示

既存のファイル管理サービスについて、Google Driveのような「整理された余白」「落ち着いた配色」「高い情報密度」「直感的な操作性」を持つUIへ全面的に改善してください。

ただし、Google Driveの画面やブランド要素をそのまま複製してはいけません。Googleのロゴ、固有アイコン、固有名称、完全に同一のレイアウトは使用せず、一般的なクラウドストレージUIとして独自性を持たせてください。

今回の主目的は、次の条件を満たす実用的なレスポンシブUIの構築です。

- デスクトップ、タブレット、スマートフォンで自然に使用できる
- フォントサイズを無駄に大きくしない
- 一方で、小さすぎて読みづらくならない
- ファイル数が多くても情報を確認しやすい
- 操作対象、現在位置、選択状態が明確である
- キーボード操作とアクセシビリティに配慮する
- 既存のバックエンド処理を壊さない
- UIだけでなく、ローディング、空状態、エラー状態まで実装する

---

# 1. 作業前の確認

実装を開始する前に、リポジトリ全体を確認してください。

最低限、次の内容を調査してください。

- 使用しているフロントエンド技術
- CSSの管理方法
- 既存コンポーネント構成
- ルーティング構成
- APIの呼び出し箇所
- ファイル・ディレクトリのデータ構造
- 認証状態の管理方法
- 既存のテスト
- lint、format、test、buildコマンド
- アイコンライブラリの有無
- ダークモード実装の有無

既存技術を尊重し、不必要なライブラリ追加や全面的な技術置換は行わないでください。

Tailwind CSSが導入されていないプロジェクトに、理由なくTailwind CSSを追加してはいけません。既存が通常のCSS、CSS Modules、SCSSなどであれば、その方式を使用してください。

作業開始前に必ず作業ブランチを作成してください。

ブランチ名の例:

```text
feat/responsive-drive-ui
```

---

# 2. デザインの基本方針

UIの方向性は、業務用途でも長時間使用できる、静かで実用的なデザインとしてください。

避けるもの:

- 巨大な見出し
- 過剰なグラデーション
- 強すぎる影
- 不必要なアニメーション
- 角丸の乱用
- 余白を取りすぎたカード
- 1画面あたりの情報量が少なすぎるデザイン
- モバイルUIをそのままデスクトップへ引き伸ばしたような構成
- すべての要素をカードで囲む設計
- アイコンだけで意味を判断させる操作
- 薄すぎる文字色
- 低コントラストな境界線

目指すもの:

- 白またはごく薄い背景
- 控えめな境界線
- 明確な選択状態
- 一貫した余白
- 視線を邪魔しない影
- ファイル名を中心とした情報設計
- 高密度だが窮屈ではない一覧
- マウス、タッチ、キーボードのすべてで操作しやすいUI

  --color-surface-subtle: #f1f3f4;
  --color-surface-hover: #eef2f7;

  --color-text-muted: #747775;

  --color-border: #dadce0;
  --color-border-strong: #bdc1c6;

  --color-primary: #2563eb;
  --color-primary-hover: #1d4ed8;
  --color-primary-soft: #dbeafe;

  --color-danger: #b3261e;

- サーフェス
- hover
- selected

# 4. タイポグラフィ

フォントは、OS標準のサンセリフを基本としてください。

```css
font-family:
  Inter,
  "Noto Sans JP",
  "Hiragino Sans",
  "Yu Gothic UI",
  "Yu Gothic",
  system-ui,
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  sans-serif;
```

InterやNoto Sans JPがプロジェクト内に存在しない場合、外部フォントを無断で追加しないでください。まずシステムフォントで実装してください。

## 基本サイズ

本文の基準サイズは16pxではなく、情報密度を考慮して14pxから15px程度を中心にしてください。

推奨値:

```css
:root {
  --font-size-xs: 0.75rem; /* 12px */
  --font-size-sm: 0.8125rem; /* 13px */
  --font-size-md: 0.875rem; /* 14px */
  --font-size-base: 0.9375rem; /* 15px */
  --font-size-lg: 1rem; /* 16px */
  --font-size-xl: 1.125rem; /* 18px */
  --font-size-2xl: 1.375rem; /* 22px */
}
```

使用方針:

- ファイル一覧のファイル名: 14pxから15px
- 補助情報: 12pxから13px
- ボタン: 13pxから14px
- ナビゲーション: 14px
- ページタイトル: 20pxから24px
- モーダルタイトル: 18pxから20px
- 空状態のタイトル: 18pxから20px
- 入力欄: 最低16px

スマートフォンの入力欄は、ブラウザによる自動ズームを避けるため16px未満にしないでください。

見出しを `clamp()` で過剰に拡大しないでください。

例:

```css
.page-title {
  font-size: clamp(1.25rem, 1.1rem + 0.5vw, 1.5rem);
}
```

最大でも24px前後に抑えてください。

本文の行間:

```css
line-height: 1.5;
```

ファイル一覧のような高密度領域では:

```css
line-height: 1.35;
```

# 5. スペーシング

4px単位を基本としたスペーシングトークンを定義してください。

```css
:root {
  --space-1: 0.25rem;
  --space-2: 0.5rem;
  --space-3: 0.75rem;
  --space-4: 1rem;
  --space-5: 1.25rem;
  --space-6: 1.5rem;
  --space-8: 2rem;
}
```

基本方針:

- ボタン内部: 縦8px前後、横12pxから16px
- 一覧行: 高さ44pxから52px
- サイドバー項目: 高さ40pxから44px
- ヘッダー: 高さ56pxから64px
- ページ左右余白: 16pxから24px
- モバイル左右余白: 12pxから16px
- カード間: 12pxから16px

余白を過剰に広く取らないでください。デスクトップでコンテンツ幅が極端に狭くならないようにしてください。

---

# 6. 角丸と影

角丸は控えめに使用してください。

```css
:root {
  --radius-sm: 6px;
  --radius-md: 10px;
  --radius-lg: 14px;
  --radius-pill: 999px;
}
```

用途:

- 入力欄: 8pxから10px
- 通常ボタン: 8pxから10px
- ファイルカード: 10pxから12px
- モーダル: 12pxから16px
- チップ: pill

すべての要素を大きな角丸にしてはいけません。

影はモーダル、メニュー、ドラッグ中の要素など、浮いている要素に限定してください。通常の一覧行やサイドバーへ強い影を付けないでください。

# 7. 全体レイアウト

デスクトップでは、次の3領域を基本構成としてください。

```text
┌──────────────────────────────────────────────┐
│ ヘッダー                                     │
├──────────────┬───────────────────────────────┤
│ サイドバー   │ メインコンテンツ              │
│              │                               │
│              │                               │
└──────────────┴───────────────────────────────┘
```

## ヘッダー

ヘッダーには次の要素を配置してください。

- サービス名またはロゴ
- モバイル用メニューボタン
- 検索欄
- 必要に応じて表示切替
- ユーザーメニュー

デスクトップ:

- 高さ60px前後
- 左側にサービス識別領域
- 中央付近に検索欄
- 右側に操作群

モバイル:

- 高さ56px前後
- サービス名を短縮可能
- 検索欄は常時展開せず、検索ボタンから展開してもよい
- 主要操作以外はメニューに格納する

ヘッダーは必要であればstickyにしてください。

```css
position: sticky;
top: 0;
z-index: 30;
```

ヘッダー背景は不透明にし、スクロール時に背後の文字が透けないようにしてください。

## サイドバー

デスクトップでは左側にサイドバーを表示してください。

推奨幅:

```css
width: 240px;
```

コンパクト表示を実装する場合:

```css
width: 72px;
```

サイドバー項目:

- マイドライブ相当のトップ
- 最近使用した項目
- お気に入り
- ゴミ箱
- 必要に応じて共有項目
- ストレージ使用量

新規作成ボタンは、ナビゲーションより上に配置してください。

モバイルではサイドバーを常時表示せず、左側から開くドロワーにしてください。

ドロワー表示中は背景にオーバーレイを表示し、Escapeキーまたは背景クリックで閉じられるようにしてください。

## メインコンテンツ

メインコンテンツには次を含めてください。

- パンくずリスト
- ページタイトル
- 操作ツールバー
- ファイル一覧
- ローディング状態
- 空状態
- エラー状態

メイン領域はサイドバーを除いた残りの幅を使用してください。

```css
min-width: 0;
```

を必ず指定し、長いファイル名による横方向のレイアウト崩れを防いでください。

# 8. パンくずリスト

現在のディレクトリ位置が分かるパンくずリストを実装してください。

要件:

- 各階層をクリックできる
- 現在位置はリンクにしなくてもよい
- 長すぎる階層は途中を省略する
- ファイル名が長い場合はellipsisを使用する
- モバイルでは親ディレクトリへ戻る操作を優先する

例:

```text
マイドライブ / プロジェクト / 2026年度 / 資料
```

モバイルでは次のような表示でも構いません。

```text
← 2026年度
```

# 9. ツールバー

一覧上部にツールバーを配置してください。

通常状態:

- 新規作成
- アップロード
- 並び替え
- 表示形式切替
- 更新
- 必要に応じてフィルター

選択状態:

- 選択件数
- ダウンロード
- 移動
- 名前変更
- 削除
- 選択解除

複数選択時は、通常ツールバーから選択用ツールバーへ明確に切り替えてください。

モバイルではすべてを横並びにせず、主要操作のみ表示し、残りは「その他」メニューへ格納してください。

---

# 10. ファイル一覧

デスクトップでは、リスト表示を基本としてください。

列の候補:

- 選択チェックボックス
- 種別アイコン
- 名前
- 更新日時
- サイズ
- 操作メニュー

例:

```text
□  📁  デザイン資料          2026/07/15 21:30   —       ⋮
□  📄  要件定義書.pdf        2026/07/14 18:20   2.4 MB  ⋮
```

## 行の仕様

- 高さ44pxから52px
- hover時に背景色を変更
- selected時に明確な背景色と境界を表示
- focus-visible時にフォーカスリングを表示
- ダブルクリックまたはEnterで開く
- チェックボックスで複数選択可能
- 行全体をクリックした際の挙動を一貫させる
- 右クリックメニューが既存仕様に合うなら実装可能
- 操作メニューは常時見えるか、hover時だけ表示する場合もキーボード操作では利用可能にする

ファイル名は最重要情報として表示してください。

```css
.file-name {
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
}
```

完全な名前はtitle属性だけに依存せず、必要に応じてツールチップまたは詳細表示で確認できるようにしてください。

## モバイル表示

モバイルではテーブルを無理に縮小しないでください。

各項目を2段構成にしてください。

```text
📄 要件定義書.pdf                    ⋮
   2026/07/14 18:20 ・ 2.4 MB
```

更新日時とサイズは補助情報として2段目に表示してください。

横スクロールを前提としたテーブルにはしないでください。

# 11. グリッド表示

既存要件に表示切替が含まれる場合は、グリッド表示も実装してください。

グリッドのカード幅:

```css
grid-template-columns: repeat(auto-fill, minmax(min(180px, 100%), 1fr));
```

カード要素:

- プレビュー領域または種別アイコン
- ファイル名
- 必要に応じて更新日時
- 操作メニュー
- 選択状態

カードの高さを不必要に大きくしないでください。

画像やPDFのサムネイルが利用できない場合は、無理にダミープレビューを生成せず、ファイル種別アイコンを使用してください。

モバイルでは2列を基本とし、非常に狭い画面では1列または小さめ2列へ調整してください。

---

# 12. 新規作成メニュー

「新規作成」ボタンから、次のようなメニューを表示してください。

- 新しいフォルダ
- ファイルをアップロード
- 必要に応じてフォルダをアップロード

メニュー要件:

- ボタンの近くへ表示
- 画面外へはみ出さない
- キーボードで上下移動可能
- Enterで決定
- Escapeで閉じる
- 外側クリックで閉じる
- 適切なARIA属性を付与

---

# 13. モーダル

次の操作にはモーダルまたは確認ダイアログを使用してください。

- フォルダ作成
- 名前変更
- 削除確認
- 移動先選択
- 必要に応じて詳細表示

モーダル要件:

- 幅は480px前後を基本
- モバイルでは画面幅から左右16pxを引いた幅
- 最大高さを超えたら内部スクロール
- 開いた際に最初の操作要素へフォーカス
- Tabキーのフォーカストラップ
- Escapeで閉じる
- 閉じた際に元の操作要素へフォーカスを戻す
- 背景スクロールを止める
- 危険操作は赤系のボタンで区別する
- Enterキーによる誤削除を避ける

削除確認では、対象名を表示してください。

例:

```text
「要件定義書.pdf」をゴミ箱へ移動しますか？
```

---

# 14. 検索

検索欄は、ヘッダー内またはコンテンツ上部に配置してください。

要件:

- placeholderを表示
- 検索アイコンを表示
- 入力内容をクリアできる
- IME入力中に不正な検索を実行しない
- 検索中状態を表示する
- 結果件数を表示する
- 結果0件の状態を実装する
- APIへの連続リクエストが発生する場合はdebounceする
- 検索中でもレイアウトが大きく動かない

検索欄の高さは40pxから44px程度としてください。

デスクトップでも極端に大きな検索バーにしないでください。

---

# 15. 状態表現

## ローディング

ローディング中に真っ白な画面を表示してはいけません。

状況に応じて以下を使用してください。

- 初回表示: スケルトン
- ボタン操作: ボタン内スピナー
- 一覧更新: 一覧領域内の控えめな進捗表示
- アップロード: 進捗バー

スケルトンは実際の一覧行に近いサイズとしてください。

## 空状態

空ディレクトリでは次を表示してください。

- 控えめなアイコンまたはイラスト
- 「このフォルダにはまだ項目がありません」
- 新規作成またはアップロードの導線

巨大なイラストや巨大な見出しは不要です。

## エラー状態

APIエラー時には次を表示してください。

- 何が失敗したか
- 再試行ボタン
- 必要に応じて詳細
- ユーザーが次に取れる行動

単に `Error` や `Something went wrong` だけを表示してはいけません。

## 通知

作成、名前変更、削除、復元などの結果はトーストで通知してください。

トースト要件:

- 画面端へ表示
- 数秒後に閉じる
- 手動で閉じられる
- エラー通知は自動で消えるまで少し長くする
- 重要なエラーはトーストだけで終わらせない

---

# 16. アップロードUI

ファイルアップロードが存在する場合、次を実装してください。

- ファイル選択
- ドラッグ&ドロップ
- アップロード中の進捗
- 成功状態
- 失敗状態
- 再試行
- 同名ファイルの扱い
- ファイルサイズ超過
- 許可されない形式
- 通信中断時の表示

ドラッグ中は、メイン領域へ明確だが過剰でないドロップゾーンを表示してください。

画面全体を強い色で覆わないでください。

---

# 17. レスポンシブ仕様

ブレークポイントはプロジェクトの既存ルールがあればそれに従ってください。存在しない場合は次を基準にしてください。

```css
/* Mobile */
@media (max-width: 639px) {
}

/* Tablet */
@media (min-width: 640px) and (max-width: 1023px) {
}

/* Desktop */
@media (min-width: 1024px) {
}
```

## 360px前後

- 横スクロールを発生させない
- サイドバーはドロワー化
- ヘッダー操作を整理
- 一覧は2段表示
- 更新日時・サイズ列を独立表示しない
- ボタン文言が収まらない場合は、アイコンと短いラベルへ変更
- タッチターゲットは最低44px程度
- モーダルは左右12pxから16pxの余白
- 固定要素がコンテンツを覆わない

## 640px前後

- サイドバーをドロワーまたはコンパクト表示
- 一覧列を必要に応じて削減
- 検索欄を表示可能
- ツールバー操作を一部メニューへ格納

## 1024px以上

- サイドバーを常時表示
- ファイル一覧の主要列を表示
- 検索欄をヘッダー内に表示
- 複数操作をツールバーへ表示

## 1440px以上

- コンテンツを無制限に横へ引き伸ばさない
- ただし、一覧画面では過剰に狭い最大幅を設定しない
- 一覧は広い画面を有効活用する
- サイドバーと列幅を極端に拡大しない

# 18. CSSレイアウト上の必須事項

次を徹底してください。

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}

html {
  min-width: 320px;
}

body {
  margin: 0;
  min-height: 100vh;
  color: var(--color-text-primary);
  background: var(--color-bg);
}

button,
input,
select,
textarea {
  font: inherit;
}

img,
svg {
  display: block;
  max-width: 100%;
}
```

FlexboxまたはGridの子要素で文字省略が必要な箇所には、適切に `min-width: 0` を設定してください。

```css
.content,
.file-name-cell,
.toolbar-main {
  min-width: 0;
}
```

`100vw` を多用しないでください。スクロールバー幅による横スクロールが発生する可能性があります。

固定幅を多用せず、`minmax()`、`clamp()`、`min()`、`max()` を適切に使用してください。

ただし、フォントサイズに過剰なviewport依存を使用しないでください。

# 19. ボタン

ボタン種別を整理してください。

- Primary
- Secondary
- Ghost
- Danger
- Icon button

通常ボタン:

```css
min-height: 40px;
padding: 0 14px;
font-size: 14px;
font-weight: 500;
```

アイコンボタン:

```css
width: 40px;
height: 40px;
```

モバイルの主要ボタンは44px程度を確保してください。

disabled状態では、単にopacityを下げるだけでなく、カーソル、背景、文字色を明確に変更してください。

アイコンだけのボタンには、必ず `aria-label` を付与してください。

# 20. フォーカス表示

すべての操作可能要素に `:focus-visible` を実装してください。

例:

```css
:focus-visible {
  outline: 3px solid color-mix(in srgb, var(--color-primary) 35%, transparent);
  outline-offset: 2px;
}
```

outlineを完全に消してはいけません。

マウス操作時には過剰に表示せず、キーボード操作時に明確に見えるようにしてください。

# 21. アクセシビリティ

最低限、次を満たしてください。

- 適切なHTML要素を使用する
- `div` のクリックだけでボタンを実装しない
- ボタンには `button`
- リンクには `a`
- 一覧が表として適切なら `table`
- ナビゲーションには `nav`
- モーダルには適切なdialog属性
- アイコンボタンにはaria-label
- 現在地にはaria-current
- 展開ボタンにはaria-expanded
- メニューには適切なロール
- エラー文を入力欄と関連付ける
- 色だけで状態を表現しない
- キーボードのみで主要操作を完結できる
- スクリーンリーダー向けの状態通知を用意する
- 最低限のコントラストを確保する

可能であれば、`prefers-reduced-motion` に対応してください。

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    scroll-behavior: auto !important;
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

# 22. アニメーション

- メニューの表示

時間は120msから200ms程度を基本としてください。

# 31. Git運用

- デフォルトブランチへ直接コミットしない
- 作業開始前に作業ブランチを作成する
- 既存の未コミット変更を勝手に削除しない
- ユーザーの変更を上書きしない
- 意味の異なる変更を1コミットへまとめすぎない
- すべての変更をコミットする
- 作業完了時に作業ツリーをクリーンにする
- 作業ブランチをoriginへpushする

コミット例:

```text
feat: add responsive application shell
feat: implement accessible file list and mobile layout
feat: add dialogs and file action states
test: cover responsive drive interactions
```
